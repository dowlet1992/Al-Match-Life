## Non-negotiable mobile UX direction

Product directive: the mobile app must be clearer, more compact, and easier to
use than the desktop web experience, with the interaction quality users expect
from products such as Instagram. It must be designed mobile-first, not rendered
as a reduced or wrapped desktop page.

- Give every screen one obvious primary purpose and keep secondary controls behind progressive disclosure.
- Prefer familiar bottom navigation and thumb-reachable actions; do not reproduce the desktop sidebar on a phone.
- Use consistent Android-native spacing, typography, hierarchy, and touch targets of at least 48 dp.
- Keep feeds, profiles, messages, notifications, calls, onboarding, and settings visually compact without hiding essential status or safety controls.
- Provide polished loading, empty, offline, error, retry, permission, and accessibility states—not only the successful state.
- Validate each screen on small and large phones, portrait/landscape, larger system fonts, light/dark themes, and RTL languages before calling it complete.
- Treat the web and mobile interfaces as two coordinated products sharing APIs and design language, not as identical layouts.
- Treat the public product name and original logo as a launch gate. The mark must be recognizable at small mobile sizes, work in color and monochrome, support an Android adaptive icon and a horizontal wordmark, and remain clearly original rather than copying Instagram or another competitor. Before design, collect 2–5 candidate names, three desired brand characteristics, target audience, preferred/forbidden colors or symbols, and any founder sketches; then perform domain, store-name, and trademark clearance before final adoption.

The Android home screen now presents equal compact follower/following counters; native lists provide bounded incremental loading, empty/error/retry states, viewer-relative labels, deduplication, and one clear follow/unfollow action per card. The remaining feed, match, chat-list, message-bubble, empty-state, field, and button measurements now use density-independent spacing with explicit 48 dp interactive minimums instead of physical pixels. Device instrumentation remains.

Profile editing now restores all seven unsaved fields across Activity recreation from a synchronized process-memory store and clears the draft on save, cancel, logout, or unauthenticated entry; no profile text enters Bundle, preferences, files, or logs. Navigation height expands for large system fonts, headings expose accessibility semantics, and async status text uses a polite live region. Android instrumentation now enables Espresso accessibility checks, verifies the primary login action at 200% system font scale, restores device settings after each test, and captures deterministic PNG candidates for screenshot baselines. JDK 17 compilation of both app and androidTest sources passes locally. - API login and account verification now return signed `Bearer` access tokens for mobile clients.
- API current-user lookup accepts `Authorization: Bearer <access_token>`.
- Web sessions continue to work for browser users.
- Access-token revocation is implemented through account session-version binding.

- Completed foundation: private messages store detected source language and per-language translation cache. Users can enable automatic translation and select `auto`/an explicit target language through privacy settings. Chat delivery translates only incoming messages, limits work to the latest batch, writes the cache once, preserves originals, and exposes provider availability. Realtime call captions remain the next translation stage.
- Started: realtime call-caption transport now accepts bounded text segments for active audio/video rooms, requires explicit per-user opt-in, verifies both room participants, rejects blocked/restricted pairs, returns only remote-speaker updates, stores no raw audio, and expires temporary segments after six hours. Caption writes are atomic: JSON uses a shared process lock and PostgreSQL locks and updates only the selected room row in one transaction. Speech recognition and translated caption presentation remain before production activation.
- Web integration: settings now include an explicit live-caption consent control. Audio/video screens provide a `CC` button, use browser speech recognition only after a user action, render interim text locally, publish final text only, poll remote captions, and stop recognition/timers when the call ends. Unsupported browsers fail gracefully without affecting the call. Server-grade speech recognition and automatic caption translation remain pending.
- Server speech fallback: when Web Speech is unavailable, the call screen records independent four-second audio fragments and sends them as bounded multipart requests. The server validates MIME/size, keeps bytes in memory only, uses the configured transcription provider, stores only the resulting text caption, and stops recording immediately when captions or the call are stopped. The default OpenAI adapter uses `v1/audio/transcriptions` with `gpt-4o-mini-transcribe` and fails closed when credentials/provider access are unavailable.
- Cost/abuse control: server transcription atomically reserves each participant sequence before any provider call. Duplicate/replayed chunks are rejected, and every speaker is limited to 18 chunks per rolling minute. JSON enforces this under the shared room lock; PostgreSQL uses the room-row transaction lock, so concurrent requests cannot bypass the limit.
- Client resilience: transcription respects `Retry-After`, ignores acknowledged duplicate chunks, retries temporary provider/network failures with exponential backoff, and suspends only local server recording after three failures. Remote-caption polling remains active so one participant's provider issue does not disable the other participant's captions.
- Upload hardening: oversized transcription requests are rejected from `Content-Length` before multipart parsing. Accepted files must match both an allowlisted MIME type and the expected audio-container signature, preventing arbitrary payloads disguised as audio from reaching provider calls.
- Consent separation: enabling live captions permits text-caption transport and local browser recognition only. Sending audio chunks to a configured server/provider requires the independent `allow_server_call_transcription` setting, which defaults off. Without it, clients may still receive remote captions but cannot upload local audio.
- Consent auditability: the first server-transcription grant and every later revocation carry separate UTC timestamps in user settings and emit dedicated security events. Re-saving an unchanged setting does not rewrite the original grant timestamp or create false audit transitions.
- Caption translation: users can independently enable automatic call-caption translation and select `auto`/a target language. Original captions render immediately; finalized remote segments are translated through a separate authorized request, cached atomically per language, and displayed only if they are still the latest segment. Provider latency or failure never blocks or replaces the original caption.
- Ephemeral-data cleanup: ending, declining, or missing a call removes all caption text, cached caption translations, and transcription reservations immediately in the same atomic room update, so a concurrent final caption write cannot survive closure.
- Signaling concurrency: offer/answer/ICE and call-state events now append through a single-room atomic operation instead of loading and rewriting the complete signaling store. JSON holds the shared store lock for the read-modify-write; PostgreSQL creates the room if needed and locks only its row. Closing state and ephemeral-caption cleanup happen in that same transaction, while full-store replacement remains isolated to administrative account cleanup.
- Signaling state safety: terminal events are idempotent. Once a room is ended, declined, or missed, duplicate terminal events and delayed answer/ICE/accepted packets are ignored under the same room lock; only a fresh offer or ringing event may reopen that deterministic participant room for a new call.
- Account cleanup isolation: deleting an account now removes only call rooms that explicitly contain that participant. New rooms maintain a normalized participant list; legacy rooms are matched through their signal history. PostgreSQL performs one targeted `DELETE`, and JSON filters under the shared store lock, so unrelated active calls are never rewritten.
- Signaling retention: room maintenance runs before pending-call discovery. Closed rooms expire after 24 hours, abandoned non-terminal rooms after seven days, and per-room signaling history after 24 hours with the existing 300-event hard cap. PostgreSQL prunes by its indexed database timestamp; JSON uses the latest signal timestamp. The accepted-call timestamp is stored separately so long-call duration remains correct after transient packets expire.
- Room reuse: a fresh offer or ringing event may reuse the deterministic room identifier after closure, but starts a new signaling session atomically. Previous transient packets and the prior `accepted_at` value are cleared, preventing stale delivery and incorrect duration attribution across separate calls.
- Network recovery: a disconnected peer receives a five-second grace period before recovery starts. The original caller alone performs ICE restart offers, avoiding negotiation glare; the receiver answers through the existing signaling channel. Recovery is capped at three attempts with increasing delays, pauses while the browser is offline, resumes on `online`, and emits one idempotent `ended` event with `connection_lost` only after the recovery budget is exhausted.
- Signaling replay safety: polling retains a one-second watermark overlap so a signal written concurrently with a room read cannot be skipped, while a bounded 600-ID client deduplication window prevents that overlap from triggering duplicate negotiation. Ended, declined, and missed states all stop media, timers, captions, and recovery before returning to chat.
- TURN connectivity: authenticated call participants fetch a private, non-cacheable ICE configuration before constructing `RTCPeerConnection`. The server requests one-hour Twilio NTS credentials, filters returned schemes and incomplete relay entries, caches them for only 50 minutes per user/room, and never exposes permanent Twilio credentials. STUN-only fallback keeps development available, while production readiness now requires the TURN provider.
- Adaptive call quality: the browser samples standard `RTCPeerConnection.getStats()` aggregates every five seconds and uploads a bounded diagnostic sample every 15 seconds. Two consecutive poor samples lower video bitrate/resolution; four good samples are required to raise it, preventing oscillation. Only RTT, jitter, packet-loss percentage, outgoing bitrate, quality class, and selected-relay use are stored, with a 24-sample room cap, atomic per-participant rate limit, and immediate purge at call closure.
- Privacy-safe quality analytics: closing a room atomically reduces temporary samples to a summary containing counts, good/fair/poor distribution, relay count, and per-metric p50/p95, then deletes every participant-linked sample. The admin-only endpoint combines room summaries into success rate, TURN rate, end reasons, distributions, and network percentiles without returning room IDs, users, timestamps, or signal details; access is security-audited.
- Signaling abuse protection: JSON signaling requests are rejected above 64 KiB before parsing. SDP and ICE use strict bounded schemas and service-state payloads use an allowlist with fixed end reasons. Per-participant offer/answer/ICE limits are enforced atomically under the JSON store or PostgreSQL row lock, while polling uses a bounded sliding window. Blocked/restricted relationships are now enforced for both signal writes and reads; rejected payload contents are never copied into audit logs.
- Cluster-wide polling limits: PostgreSQL production mode stores only a SHA-256 limiter key, category, fixed-window epoch, count, and expiry in `rate_limit_buckets`. Atomic upsert makes the 120/minute poll quota consistent across workers and instances; expired buckets are deleted transactionally. JSON development mode retains the bounded in-process sliding window and creates no runtime file.
- Atomic call state machine: signaling now enforces ringing → offer/accepted → answer/ICE → terminal transitions under the JSON lock or PostgreSQL room-row lock, including caller/recipient direction. Empty-room terminal events, answers without offers, duplicate accepts, client-generated missed events, and non-ringing reopen attempts are rejected. The browser queues local trickle-ICE until its corresponding SDP has been persisted, preventing strict validation from dropping candidates generated during `setLocalDescription` or ICE restart.
- Idempotent signaling delivery: each logical browser signal carries one 16–80 character URL-safe event ID. Network and temporary server failures retry the identical serialized request up to three times; the server checks that ID under the room lock before transition/rate-limit logic and returns an explicit duplicate acknowledgment. Lost HTTP responses therefore cannot create repeated SDP/ICE, terminal state, quota usage, or call-history events.
- Recipient delivery acknowledgments: processed remote events are ACKed in bounded batches under the room lock, and only the addressed recipient can mutate delivery state. Unacknowledged events bypass the time cursor and are redelivered; an ACK lost in transit is retried idempotently without changing its original timestamp. The browser records an event in its bounded dedupe set only after `handleSignal` succeeds and attempts terminal ACK before closing media/navigation.
- Server call deadlines: signaling poll maintenance atomically converts unanswered ringing to missed after 45 seconds and accepted-but-unanswered negotiation to ended/`negotiation_timeout` after 30 seconds. It uses deterministic timeout event IDs, performs the same ephemeral-data purge as manual closure, and emits history only for the request that performed the transition. This removes the former dependency on opening a chat to discover missed calls; answered calls are never expired by negotiation maintenance.
- Background call maintenance now has a production-safe, one-shot worker with dry-run default, explicit `--apply`, bounded batches, PII-free output, and PostgreSQL `SKIP LOCKED` coordination. Scheduling it every minute makes missed-call and negotiation deadlines independent of client polling and web-worker lifecycle.
- Authenticated push-device lifecycle is now available for Android/FCM, iOS/APNs, and Web Push: strict registration validation, atomic token ownership, safe device listing without token disclosure, per-device revocation, account-deletion cleanup, PostgreSQL constraints, and a JSON development fallback. Provider delivery credentials and incoming-call dispatch remain the next deployment stage; the API does not pretend delivery succeeded before a provider is configured.
- Incoming-call transactional outbox: the first valid `ringing` signal now persists its push job in the same JSON lock or PostgreSQL transaction as signaling. Signal retries reuse the outbox primary key and cannot duplicate delivery jobs. Jobs carry a 45-second expiry, bounded retry state, and no provider token; the delivery worker will resolve active devices only when claiming a job.
